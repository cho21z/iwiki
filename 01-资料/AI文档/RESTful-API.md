## 什么是 RESTful API

如果你刚接触后端开发，几乎一定会频繁听到两个词：`REST` 和 `RESTful API`。

很多新人一开始会把它理解成：

- 用 HTTP 写接口
- 用 JSON 传数据
- 用 `GET/POST/PUT/DELETE` 区分操作

这些理解**不算错，但都只是表象**。

更准确地说：

**RESTful API 是一种围绕“资源（Resource）”来设计 Web 接口的风格。**

它的核心目标不是“把接口写得像某种固定格式”，而是：

- 让接口语义清晰
- 让前后端协作成本更低
- 让系统更容易理解、扩展、维护
- 让 API 更符合 HTTP 协议本身的设计方式

---

## 一、先理解 REST 是什么

`REST` 全称是 `Representational State Transfer`，通常翻译为**表述性状态转移**。

这个名字听起来很学术，但落到工程实践里，你可以先记住一句最重要的话：

**REST 关注的是“资源”以及“如何通过统一的方式操作资源”。**

在后端系统里，很多东西都可以被抽象成资源，比如：

- 用户
- 订单
- 商品
- 购物车
- 评论
- 文章
- 权限关系

也就是说，RESTful API 不是围绕“动作”设计，而是围绕“资源”设计。

### 一个最直观的对比

假设我们要做“用户管理”接口。

不太 RESTful 的写法可能是：

```http
POST /createUser
POST /deleteUser
POST /updateUser
GET  /getUserById?id=1
```

RESTful 风格更倾向于这样：

```http
POST   /users
GET    /users/1
PUT    /users/1
DELETE /users/1
```

两者都能完成功能，但第二种有几个明显优势：

- URL 表示的是**资源是什么**，而不是**你打算做什么动作**
- 动作交给 HTTP 方法表达，语义更统一
- 一眼就能看懂资源模型
- 更容易形成团队约定，降低沟通成本

---

## 二、RESTful API 的核心思想

### 1. 一切面向资源

RESTful API 首先要回答的问题不是“我要写几个接口”，而是：

**我的系统里有哪些核心资源？**

举个电商系统的例子，可能存在这些资源：

- `users`
- `products`
- `orders`
- `carts`
- `payments`

然后再思考：

- 如何创建资源
- 如何查询资源
- 如何修改资源
- 如何删除资源
- 资源之间是什么关系

### 2. 用 HTTP 方法表达动作

在 RESTful API 中，URL 应该尽量只表达资源定位，而具体操作交给 HTTP Method。

常见映射如下：

| HTTP Method | 含义 | 典型用途 |
|---|---|---|
| `GET` | 获取资源 | 查询列表、查询详情 |
| `POST` | 创建资源 | 新增记录、触发创建型操作 |
| `PUT` | 整体更新资源 | 完整覆盖更新 |
| `PATCH` | 部分更新资源 | 修改部分字段 |
| `DELETE` | 删除资源 | 删除指定资源 |

例如：

```http
GET    /orders
GET    /orders/1001
POST   /orders
PATCH  /orders/1001
DELETE /orders/1001
```

### 3. URL 表达资源层级

URL 最重要的职责是：**定位资源**。

例如：

```http
/users
/users/1
/users/1/orders
/orders/1001/items
```

这些 URL 体现了资源及其关系：

- `/users`：用户集合
- `/users/1`：ID 为 1 的用户
- `/users/1/orders`：该用户的订单集合
- `/orders/1001/items`：某个订单下的商品项

### 4. 无状态（Stateless）

REST 强调**服务端无状态**。

意思是：

**每次请求都应该携带完成本次处理所需的全部信息，服务端不依赖“上一次请求的上下文状态”来理解这次请求。**

例如认证时，常见做法是：

- 请求头里带上 Token
- 服务端基于 Token 识别用户身份

而不是：

- 第一次请求记住你是谁
- 第二次请求默认沿用上次状态

无状态带来的好处：

- 更利于横向扩展
- 更容易做负载均衡
- 更方便排查问题
- 更适合微服务和云原生架构

### 5. 统一接口

REST 强调统一接口风格，背后意思是：

**不同资源，尽量遵循同一种设计语言。**

比如：

- 查列表都用 `GET /resources`
- 查详情都用 `GET /resources/{id}`
- 创建都用 `POST /resources`
- 删除都用 `DELETE /resources/{id}`

这样前端、测试、客户端同学拿到接口文档后，学习成本会显著下降。

## 三、怎么理解“资源”

新人最容易卡住的问题是：

**到底什么算资源？**

一个简单判断方法：

如果某个业务对象可以被唯一标识、可以被创建/查询/修改/删除，那么它通常就是资源。

例如：

- 用户是资源
- 订单是资源
- 商品是资源
- 评论是资源

但像下面这些，更像动作而不是资源：

- `login`
- `refreshToken`
- `sendSms`
- `payOrder`

这类场景并不是不能设计 RESTful，而是要进一步抽象。

例如：

- 登录可以理解成创建一个会话资源：`POST /sessions`
- 刷新 Token 可以理解成更新令牌资源
- 发送短信可以理解成创建一个短信发送任务
- 支付订单可以理解成创建一个支付资源：`POST /payments`

这也是为什么 RESTful 设计并不只是背几个 URL 模板，而是训练你从**动作思维切换到资源思维**。

## 四、RESTful API 的 URL 设计建议

### 1. 用名词，不用动词

推荐：

```http
GET /users
POST /orders
DELETE /products/10
```

不推荐：

```http
GET /getUsers
POST /createOrder
POST /deleteProduct
```

原因很简单：

- URL 负责标识资源
- HTTP Method 负责表达动作

### 2. 集合用复数

一般建议资源集合使用复数形式：

```http
/users
/orders
/products
```

比起：

```http
/user
/order
/product
```

复数更符合“资源集合”的语义，也更常见。

### 3. 层级不要过深

例如：

```http
/users/1/orders/1001/items/3
```

这种不是绝对不能用，但当层级太深时，会暴露出两个问题：

- URL 难读
- 资源关系耦合过重

通常建议层级控制在 2 到 3 层内，过深时可以考虑独立建模。

### 4. 用连字符或统一风格

如果资源名由多个单词组成，建议统一规范，例如：

```http
/order-items
/user-profiles
```

或者团队统一约定下划线：

```http
/order_items
/user_profiles
```

**关键不是哪种一定最好，而是全项目保持一致。**

### 5. 不把查询条件写进路径语义里

推荐：

```http
GET /orders?status=paid&user_id=1&page=1&page_size=20
```

不推荐：

```http
GET /getPaidOrdersByUserId/1/1/20
```

查询参数天然适合表达：

- 过滤条件
- 排序
- 分页
- 搜索关键词

## 五、HTTP 方法的正确使用

### 1. `GET`：获取资源

用于读取数据，不应该产生副作用。

示例：

```http
GET /users
GET /users/1
GET /orders?status=paid
```

注意：

- `GET` 应该是安全的（safe）
- 一般不应该因为一次 `GET` 就修改数据库数据
- 不要把“删除”“下单”“扣库存”这种动作塞进 `GET`

错误示例：

```http
GET /deleteOrder?id=1001
```

这会导致语义混乱，甚至可能被缓存、预加载、爬虫等误触发。

### 2. `POST`：创建资源

常用于新增资源，也常用于那些**不适合简单映射为幂等更新**的创建型操作。

示例：

```http
POST /users
POST /orders
POST /payments
```

请求体通常包含创建所需字段，例如：

```json
{
  "name": "Alice",
  "email": "alice@example.com"
}
```

### 3. `PUT`：整体更新资源

`PUT` 更强调**完整替换**。

例如：

```http
PUT /users/1
```

请求体可能是该资源最新的完整表示：

```json
{
  "name": "Alice",
  "email": "alice_new@example.com",
  "status": "active"
}
```

如果你的系统并不区分“全量更新”和“部分更新”，很多团队也会直接用 `PUT` 做更新，但要在接口文档中写清楚语义。

### 4. `PATCH`：部分更新资源

如果只是更新几个字段，`PATCH` 往往更贴切。

```http
PATCH /users/1
```

```json
{
  "status": "disabled"
}
```

这比用 `PUT` 传整个对象更轻量，也更明确。

### 5. `DELETE`：删除资源

用于删除指定资源。

```http
DELETE /users/1
DELETE /orders/1001
```

这里还要注意一个现实问题：

很多业务并不是真删，而是逻辑删除。即使底层是逻辑删除，API 语义层面仍然可以使用 `DELETE`，因为它表达的是“该资源对外被移除”。

## 六、什么是幂等性，为什么 RESTful 常提它

后端工程里，**幂等性**是非常重要的概念。

所谓幂等，可以简单理解为：

**同一个请求执行一次和执行多次，结果一致。**

例如：

- `GET /users/1` 调 10 次，都是获取同一个用户
- `DELETE /users/1` 调多次，最终效果仍然是“这个资源被删除”
- `PUT /users/1` 用同样的数据重复调用，结果应该一致

而 `POST /orders` 通常不是天然幂等的，因为重复提交可能创建多个订单。

### 常见幂等性分类

| 方法 | 是否通常幂等 | 说明 |
|---|---|---|
| `GET` | 是 | 查询不应改变资源状态 |
| `PUT` | 是 | 相同请求重复执行结果一致 |
| `DELETE` | 是 | 删除多次，资源最终都不存在 |
| `POST` | 通常否 | 可能重复创建资源 |
| `PATCH` | 不一定 | 取决于补丁语义 |

### 为什么要关注幂等

因为真实生产环境里经常会发生：

- 客户端超时重试
- 网关重复投递
- 用户重复点击提交按钮
- 消息系统重复消费

如果接口没有幂等设计，就可能出现：

- 重复下单
- 重复扣款
- 重复发券
- 重复发消息

所以 RESTful API 设计时，不仅要关心 URL 好不好看，更要关心接口语义和系统行为是否可靠。

## 七、状态码怎么用

很多新人写接口时，喜欢所有请求都返回 HTTP `200`，然后在响应体里自己放一套“成功失败码”。

这不是完全不能做，但从 RESTful 的角度看，**HTTP 状态码本身就应该承担语义表达责任**。

常见状态码建议如下：

| 状态码 | 含义 | 典型场景 |
|---|---|---|
| `200 OK` | 请求成功 | 查询成功、更新成功 |
| `201 Created` | 创建成功 | 新建用户、新建订单 |
| `204 No Content` | 成功但无返回体 | 删除成功、某些更新成功 |
| `400 Bad Request` | 请求参数错误 | 参数缺失、格式错误 |
| `401 Unauthorized` | 未认证 | 未登录、Token 无效 |
| `403 Forbidden` | 已认证但无权限 | 没有访问权限 |
| `404 Not Found` | 资源不存在 | 用户不存在、订单不存在 |
| `409 Conflict` | 资源冲突 | 重复创建、版本冲突 |
| `422 Unprocessable Entity` | 语义校验失败 | 业务字段校验不通过 |
| `500 Internal Server Error` | 服务端异常 | 未预期系统错误 |

### 举几个例子

#### 创建用户成功

```http
POST /users
```

返回：

- 状态码：`201 Created`
- 响应体：新创建的用户信息

#### 查询一个不存在的订单

```http
GET /orders/999999
```

返回：

- 状态码：`404 Not Found`

#### 参数格式错误

```http
GET /users?page=-1
```

返回：

- 状态码：`400 Bad Request`

### 一个经验建议

**业务码可以有，但不要替代 HTTP 状态码。**

比较合理的做法是：

- HTTP 状态码表示协议层结果
- 业务码表示更细粒度业务语义

例如：

```json
{
  "code": "USER_NOT_FOUND",
  "message": "用户不存在"
}
```

同时 HTTP 状态码为 `404`。

## 八、请求体、响应体该怎么设计

### 1. 请求体表达“客户端想提交什么”

创建订单：

```json
{
  "user_id": 1001,
  "items": [
    {
      "product_id": 2001,
      "quantity": 2
    }
  ],
  "address_id": 3001
}
```

### 2. 响应体表达“服务端处理后的结果”

```json
{
  "id": 9001,
  "user_id": 1001,
  "status": "pending",
  "total_amount": 199.00,
  "created_at": "2026-03-27T14:00:00Z"
}
```

### 3. 字段命名保持统一

推荐统一风格，例如：

- 全部使用 `snake_case`
- 或全部使用 `camelCase`

不要同一个系统里出现：

```json
{
  "user_id": 1,
  "userName": "Tom",
  "createdAt": "2026-03-27"
}
```

这种风格混用会显著增加沟通与维护成本。

### 4. 错误响应要稳定

不要一会儿返回：

```json
{
  "msg": "参数错误"
}
```

一会儿又返回：

```json
{
  "error_message": "非法请求"
}
```

建议统一错误结构，例如：

```json
{
  "code": "INVALID_PARAMETER",
  "message": "page 必须大于 0",
  "details": {
    "field": "page"
  }
}
```

## 九、分页、过滤、排序、搜索怎么设计

这是业务接口里非常常见的一类需求。

### 1. 分页

常见写法：

```http
GET /users?page=1&page_size=20
```

### 2. 过滤

```http
GET /orders?status=paid
GET /products?category=book
```

### 3. 排序

```http
GET /orders?sort_by=created_at&sort_order=desc
```

### 4. 搜索

```http
GET /users?keyword=tom
```

### 5. 组合使用

```http
GET /orders?status=paid&page=1&page_size=20&sort_by=created_at&sort_order=desc
```

### 一点实践建议

分页接口除了列表数据，通常还会返回一些分页元信息，例如：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 125,
  "has_more": true
}
```

这样前端更容易渲染分页组件。

## 十、一个完整的 RESTful 设计示例

下面以“文章系统”为例，看看如何设计一组相对规范的 RESTful API。

### 资源定义

核心资源：

- 文章：`articles`
- 评论：`comments`
- 标签：`tags`

### 接口设计

#### 1. 获取文章列表

```http
GET /articles?page=1&page_size=10
```

#### 2. 获取文章详情

```http
GET /articles/123
```

#### 3. 创建文章

```http
POST /articles
```

请求体：

```json
{
  "title": "什么是 RESTful API",
  "content": "这是文章内容",
  "tag_ids": [1, 2, 3]
}
```

#### 4. 更新文章

```http
PATCH /articles/123
```

请求体：

```json
{
  "title": "RESTful API 入门"
}
```

#### 5. 删除文章

```http
DELETE /articles/123
```

#### 6. 获取文章评论

```http
GET /articles/123/comments
```

#### 7. 给文章新增评论

```http
POST /articles/123/comments
```

### 这种设计的优点

- 资源边界清晰
- URL 可读性高
- 接口风格统一
- 前后端更容易协作
- 后期扩展成本更低

## 十一、RESTful API 不等于“机械套模板”

这点非常重要。

很多人学 RESTful，到最后变成了背模板：

- 列表就是 `GET /xxx`
- 详情就是 `GET /xxx/{id}`
- 创建就是 `POST /xxx`

模板当然重要，但真正成熟的后端工程师会继续往下思考：

- 资源抽象是否合理
- 语义是否稳定
- 状态码是否正确
- 是否满足幂等要求
- 是否便于缓存
- 是否适合权限控制
- 是否方便审计和监控
- 是否兼容未来扩展

换句话说：

**RESTful 是设计思想，不只是接口命名技巧。**

---

## 十二、常见误区

### 误区 1：只要用了 JSON 就是 RESTful

不是。

JSON 只是数据交换格式。你就算返回 JSON，也可能把接口设计得非常混乱。

### 误区 2：所有接口都必须严格 RESTful

也不是。

工程实践里，RESTful 是主流设计风格，但不是宗教。

有些场景天然更偏“动作”：

- 登录
- 导出报表
- 批量审批
- 发起支付
- 上传文件分片合并

这些场景有时需要在 RESTful 风格和工程可用性之间做平衡。

正确做法通常是：

**优先遵循 RESTful 思路，在确实不合适时做适度折中，并保持团队约定一致。**

### 误区 3：接口路径越短越好

不一定。

路径设计应该优先追求：

- 语义明确
- 风格统一
- 易于理解

不是单纯追求“短”。

### 误区 4：统一返回 `200` 最省事

短期看是省事，长期看会导致：

- 网关层难处理
- 客户端难区分错误类型
- 监控系统难统计
- 排障效率低

所以应该合理使用 HTTP 状态码。

### 误区 5：URL 里写动作完全不行

也不是绝对。

例如：

```http
POST /orders/1001/cancel
POST /orders/1001/confirm
```

这种“动作型子资源”在很多业务场景里是可以接受的，特别是当这个动作难以自然映射为标准 CRUD 时。

关键是：

- 不要滥用
- 要有明确业务语义
- 团队内保持统一约定

## 十三、什么时候可以不那么“纯 RESTful”

真实项目里，我们追求的是**合理**，不是**教条**。

以下场景经常会做一些折中：

### 1. 登录与认证

例如：

```http
POST /login
```

这不是最“纯”的 RESTful 设计，但非常直观。

如果你要更资源化一些，也可以设计为：

```http
POST /sessions
```

两种都有人用，关键看团队约定和系统风格。

### 2. 批量操作

例如批量删除用户：

```http
POST /users/batch-delete
```

或者：

```http
DELETE /users?ids=1,2,3
```

哪种更好，没有绝对答案，要结合：

- 网关能力
- 框架支持
- 审计需求
- 可读性

### 3. 复杂业务动作

例如订单取消、审核通过、发布文章，这些都带有显著业务动作。

此时可以采用：

```http
POST /orders/1001/cancel
POST /articles/123/publish
POST /reviews/888/approve
```

这类设计在工业界非常常见。

## 十四、作为后端工程师，设计 RESTful API 时要考虑什么

新人常把接口开发理解成“把功能做出来”。

但站在一个成熟后端工程师的角度，设计 API 时至少要考虑下面这些问题：

### 1. 资源模型是否清晰

- 核心资源是什么
- 从属资源是什么
- 资源间关系是否合理

### 2. 接口语义是否稳定

- URL 是否表达资源，而不是零散动作
- HTTP 方法是否使用正确
- 状态码是否符合语义

### 3. 是否便于前端和客户端使用

- 字段命名是否统一
- 错误结构是否稳定
- 分页与过滤是否清晰
- 文档是否容易理解

### 4. 是否便于系统演进

- 后续新增字段是否兼容
- 是否需要版本控制
- 是否考虑幂等和重试
- 是否适合缓存

### 5. 是否具备生产可用性

- 权限校验是否明确
- 限流策略是否有考虑
- 审计日志是否可追踪
- 错误码是否便于监控告警

也就是说，API 设计不是“写 controller”那么简单，它本质上是业务建模能力和工程抽象能力的体现。

## 十五、给新人的一套 RESTful API 设计口诀

如果你暂时还记不住那么多内容，可以先记住下面这套口诀：

- **先想资源，再想接口**
- **URL 用名词，Method 表动作**
- **列表和详情分清楚**
- **过滤、分页、排序放查询参数**
- **合理使用状态码，不要全返回 200**
- **注意幂等，防止重复请求出问题**
- **错误响应结构保持统一**
- **风格一致比“自认为高级”更重要**

## 十六、一个新人最容易上手的设计模板

假设你拿到一个新资源 `books`，可以先按下面模板思考：

```json
}
```

当你能把一个资源按这个模板设计清楚，你就已经迈出了 RESTful API 设计的第一步。

## 十七、最后总结

RESTful API 本质上是一种**面向资源的接口设计风格**。

它希望我们：

- 用 URL 表达资源
- 用 HTTP Method 表达操作
- 用状态码表达结果
- 用统一风格降低沟通成本
- 用无状态和幂等思维提高系统可靠性

对于新人来说，学习 RESTful API 的关键不在于死记硬背，而在于逐步建立这几个意识：

- **资源意识**：先想“对象是什么”
- **语义意识**：接口名、方法、状态码要一致
- **工程意识**：接口不是写给自己看的，是写给团队和未来维护者看的
- **可靠性意识**：重复请求、异常返回、权限控制都要提前考虑

当你开始从“写一个能用的接口”，成长到“设计一组可维护、可扩展、可协作的接口”，你就真正开始进入后端工程实践了。

## 十八、课后思考题

建议新人在看完这份讲义后，自己尝试回答下面几个问题：

1. 为什么说 RESTful API 是围绕资源，而不是围绕动作设计？
2. `GET /getUserById?id=1` 和 `GET /users/1`，你更推荐哪一个？为什么？
3. `PUT` 和 `PATCH` 有什么区别？
4. 为什么不能把“删除订单”设计成 `GET /deleteOrder?id=1`？
5. 为什么生产环境里需要重视幂等性？
6. 为什么不建议所有接口都返回 `200`？
7. 登录、支付、审核这类动作，怎样在 RESTful 风格下做合理设计？

如果这些问题你能说清楚 70%，说明你已经对 RESTful API 有了比较扎实的入门理解。
