> 💡 为什么不能所有请求都返回 200？HTTP 状态码和业务码该怎么配合使用？

不用所有的请求都返回 HTTP 200，HTTP 的状态码本身应该承担起语义表达责任。

## 常见状态码
| Status Code | 含义 | 场景 |
| --- | --- | --- |
| 200 OK | 请求成功 | 查询成功、更新成功 |
| 201 Created | 创建成功 | 新建用户、新订单 |
| 204 No Content | 请求成功，但无内容返回 | 删除成功、某些更新成功 |
| 400 Bad Request | 请求参数错误 | 参数缺失、格式错误 |
| 401 Unauthorized | 请求未授权 | 未认证、Token 无效 |
| 403 Forbidden | 已认证但无权限 | 没有访问权限 |
| 404 Not Found | 资源不存在 | 用户不存在、订单不存在 |
| 409 Conflict | 资源冲突 | 重复创建 |
| 422 Unprocessable Entity | 语义校验失败 | 业务字段校验不通过 |
| 500 Internal Server Error | 服务器错误 | 未预期系统错误 |

## HTTP 状态码 与 业务码
业务码可以有，但不要替代 HTTP 状态码。比较合理的做法是：
- HTTP 状态码表示协议层结果
- 业务码表示更细粒度业务语义

例如一个用户不存在的响应，HTTP 状态码为 404，响应体再用业务码表达更细的语义：
```json
{
  "code": "USER_NOT_FOUND",
  "message": "用户不存在"
}
```
