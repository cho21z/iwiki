# 项目实践

> 记录完整项目的设计思路、实现过程、踩坑与复盘。这里不放零散的知识点，只放"完整可交付"的项目。

---

## 📍 这里和其他目录的区别

| 维度 | 其他目录（如 `go/`、`distributed/`） | `project/`（这里） |
|---|---|---|
| 单位 | 一个**问题** | 一个**项目** |
| 粒度 | 单个知识点 | 端到端完整工程 |
| 格式 | 问题驱动（简短回答 + 展开 + 我的理解） | 项目复盘（背景 + 架构 + 实现 + 复盘） |
| 文件命名 | 中文问题句 | 中文项目名（如 `秒杀系统设计与实现.md`） |
| 价值 | 检验对某个点的理解 | 串联多个领域的知识，形成系统性能力 |

## 🎯 推荐项目方向

按由浅入深的顺序排列，每个项目都能串联多个领域的知识，**做完后可以反过来补充对应领域的问题卡片**。

### 项目 1：短链接服务（入门级）

**串联领域**：`go` + `database` + `cache` + `system-design`

**核心挑战**：
- 短码生成算法（自增 ID + Base62 / 哈希 / 雪花 ID）
- 高并发读取的缓存设计
- 防刷与限流
- 短链 301/302 跳转的 SEO 影响

**推荐产出文件名**：`短链接服务设计与实现.md`

---

### 项目 2：高并发秒杀系统（进阶级）

**串联领域**：`go` + `cache` + `mq` + `database` + `distributed` + `system-design`

**核心挑战**：
- Redis 扣减库存（Lua 原子操作）
- MQ 异步下单削峰
- MySQL 最终落库与一致性

- 限流与降级（熔断、令牌桶）
- 压测与性能分析

**推荐产出文件名**：`高并发秒杀系统设计与实现.md`

---

### 项目 3：配置中心 / 策略平台（中高级）

**串联领域**：`go` + `database` + `distributed` + `cloud-native`

**核心挑战**：

- 权限与审计

**推荐产出文件名**：`配置中心设计与实现.md`

---

### 项目 4：分布式调度系统（高级）

**串联领域**：`go` + `database` + `distributed` + `cloud-native` + `system-design`

**推荐产出文件名**：`分布式调度系统设计与实现.md`

## 📝 项目笔记建议结构

项目类笔记不严格遵循问题驱动格式，建议包含以下章节：

```markdown
# [项目名称]

## 1. 项目背景与目标
- 为什么做这个项目
- 解决什么问题
- 预期达到什么指标

## 2. 架构设计
- 整体架构图（推荐用 Mermaid / PlantUML）
- 关键技术选型与理由
- 数据流与调用链

## 3. 核心模块说明
- 每个模块的职责
- 关键接口与数据结构

## 4. 踩坑与解决方案
- 遇到的真实问题
- 如何排查、如何解决

## 5. 性能数据与优化记录
- 压测数据（QPS、延迟、资源占用）
- 优化前后对比

## 6. 复盘与改进方向
- 当前方案的局限
- 如果重做会怎么改

## 7. 关联问题卡片
- [问题 A](../go/xxx.md)
- [问题 B](../distributed/xxx.md)
```

## 💡 使用建议

1. **不必按顺序做完所有项目**——选最感兴趣或最贴合工作场景的开始
2. **做完一个项目后回头看**——你会发现之前学过的问题卡片有了新的理解，可以回去补充"我的理解"章节
3. **项目可以迭代**——v1 实现基础功能，v2 加上压测，v3 加上可观测性，每次迭代都是一次深度学习
4. **失败的尝试也值得记录**——在踩坑章节如实写下来，比"完美方案"更有价值
